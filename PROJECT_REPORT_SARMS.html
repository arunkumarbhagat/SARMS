1. Project Overview

The Student Attendance & Result Management System (SARMS) is a full-stack university portal designed to digitize and centralize the management of student academic records. It replaces manual paper-based attendance registers and mark sheets with a structured, role-based web application accessible to Students, Teachers, and Administrators.

The system provides each role with a dedicated dashboard tailored to their responsibilities. Students can view their attendance records, subject marks, and enrolled subjects. Teachers can mark attendance for any student, enter or update marks, and view the complete student list. Administrators can add new students, teachers, and subjects to the system, with automatic login account creation for each new entity.

The backend is built with Python Flask using a Blueprint-based MVC architecture. Data is stored in MongoDB, and the frontend uses Bootstrap 5 for a clean, responsive interface with tabbed dashboards.

2. Objectives

  • Build a centralized university portal to manage student attendance and academic results
  • Implement role-based access control for three distinct roles: Student, Teacher, and Admin
  • Allow teachers to mark attendance per student, per subject, per date with Present/Absent status
  • Allow teachers to enter and update marks per student per subject using MongoDB upsert
  • Allow students to view their own attendance records, marks, and enrolled subjects
  • Allow admins to add students, teachers, and subjects through a single dashboard
  • Automatically create login credentials when admin adds a student or teacher
  • Protect all dashboards with session-based role guards to prevent unauthorized access
  • Provide a responsive, tab-based UI using Bootstrap 5 for all three dashboards
  • Seed the database with realistic sample data for demonstration and testing

4. System Architecture

SARMS follows the MVC (Model-View-Controller) architectural pattern implemented through Flask Blueprints. Each blueprint handles a specific domain of the application.

Layer Component Responsibility
Model models/ (student_model, teacher_model, attendance_model, subject_model) Define document schemas and data structure functions
View templates/ (Jinja2 HTML templates) Render HTML pages with dynamic data injected from Flask context
Controller routes/ (auth, student, teacher, admin blueprints) Handle HTTP requests, apply business logic, query MongoDB, return JSON or render templates
Config config.py MongoDB connection and all 6 collection references
Entry Point app.py Flask app initialization, blueprint registration, dashboard routes with session guards

6. Database Design

SARMS uses MongoDB database named sarms_local with 6 collections. All collections are defined and accessed centrally through config.py.

6.1 users Collection

Stores login credentials for all three roles. Student and teacher accounts are auto-created when admin adds them. Admin account is created via seed_admin.py.

Field Type Description
_id ObjectId Auto-generated MongoDB document ID
username String Login username (equals student_id or teacher_id for auto-created accounts)
password String Login password (plain text; equals username for auto-created accounts)
role String "student", "teacher", or "admin"
ref_id String Links to student_id or teacher_id; stored in session as user_id after login

6.2 students Collection

Stores academic profile information for each student.

Field Type Description
_id ObjectId Auto-generated MongoDB document ID
student_id String Unique student identifier (e.g., "23BAI001")
name String Full name of the student
department String Academic department (e.g., "Computer Science")
semester Integer Current semester number (1–8)

6.3 teachers Collection

Stores profile information for each teacher.

Field Type Description
_id ObjectId Auto-generated MongoDB document ID
teacher_id String Unique teacher identifier (e.g., "TCH001")
name String Full name with title (e.g., "Dr. Ramesh Kumar")
department String Department the teacher belongs to

6.4 subjects Collection

Stores subject definitions with semester and teacher assignment.

Field Type Description
_id ObjectId Auto-generated MongoDB document ID
subject_code String Unique subject code (e.g., "CS301")
subject_name String Full subject name (e.g., "Data Structures")
semester Integer Semester in which the subject is taught
teacher_id String Reference to the assigned teacher's teacher_id

6.5 attendance Collection

Stores individual attendance records per student, per subject, per date. Multiple records can exist for the same student and subject across different dates.

Field Type Description
_id ObjectId Auto-generated MongoDB document ID
student_id String Reference to the student's student_id
subject_code String Reference to the subject's subject_code
status String "Present" or "Absent"
date String Date of the class in YYYY-MM-DD format

6.6 marks Collection

Stores marks per student per subject. Uses upsert so re-entering marks for the same student+subject updates the existing record rather than creating a duplicate.

Field Type Description
_id ObjectId Auto-generated MongoDB document ID
student_id String Reference to the student's student_id
subject_code String Reference to the subject's subject_code
marks Number Marks obtained (0–100)

7. Module Description

7.1 Authentication Module — routes/auth_routes.py

Handles role-aware login using a JSON API consumed by JavaScript fetch on the frontend. The login page is rendered with the role context so it can display the appropriate heading. On successful login, the user's username, role, and ref_id (student_id or teacher_id) are stored in the Flask session.

Function Route Method Description
login_page(role) /login/<role> GET Renders login.html with the role variable injected into the template for conditional heading display
login() /login POST (JSON) Queries users_collection with username + password; sets session on success; returns redirect URL as JSON; returns 401 on failure

7.2 Student Module — routes/student_routes.py

Provides data APIs for the student dashboard. The /student/data endpoint uses the user_id from the session (which equals the student_id) to fetch only that student's records.

Function Route Method Description
get_students() /students GET Returns all students as a JSON list (used by admin and teacher views)
get_my_data() /student/data GET Returns the logged-in student's attendance, marks, and all subjects as a single JSON object; returns 401 if not logged in

7.3 Teacher Module — routes/teacher_routes.py

Provides APIs for teacher operations. Both write operations validate that the target student exists before inserting or updating records. Marks use MongoDB upsert to prevent duplicate entries.

Function Route Method Description
mark_attendance() /teacher/mark_attendance POST (JSON) Validates student_id exists; inserts attendance document with student_id, subject_code, status, date; returns 400 if student not found
enter_marks() /teacher/enter_marks POST (JSON) Validates student_id exists; upserts marks document using {student_id, subject_code} as filter and $set {marks} as update; returns 400 if student not found
get_students() /teacher/students GET Returns all students as a JSON list for the teacher's student list tab

7.4 Admin Module — routes/admin_routes.py

Provides APIs for all admin management operations. Adding a student or teacher automatically creates a corresponding login account in the users collection using upsert, with the entity ID as both username and default password.

Function Route Method Description
add_student() /admin/add_student POST (JSON) Inserts student document; upserts user login with role="student", username=student_id, password=student_id, ref_id=student_id
add_teacher() /admin/add_teacher POST (JSON) Inserts teacher document; upserts user login with role="teacher", username=teacher_id, password=teacher_id, ref_id=teacher_id
add_subject() /admin/add_subject POST (JSON) Inserts subject document with subject_code, subject_name, semester, teacher_id
get_students() /admin/students GET Returns all students from students_collection as JSON list
get_teachers() /admin/teachers GET Returns all teachers from teachers_collection as JSON list

7.5 Dashboard Routes — app.py

Core dashboard routes are defined directly in app.py with inline session role guards. Each dashboard route checks the session role and redirects to home if the role does not match.

Route Guard Template
/student_dashboard session.role == "student" student_dashboard.html
/teacher_dashboard session.role == "teacher" teacher_dashboard.html
/admin_dashboard session.role == "admin" admin_dashboard.html
/logout None Clears session, redirects to /

9. User Interface

9.1 Home Page — home.html

The landing page presents three role selection cards side by side using Bootstrap's grid system. Each card displays a role emoji, title, description, and a colored login button. Student uses blue, Teacher uses green, and Admin uses red. The page uses a light grey background with white cards and subtle box shadows. Clicking a card navigates to /login/<role>.

9.2 Login Page — login.html

A centered Bootstrap card form with username and password fields. The page heading dynamically shows the role (Student Login / Teacher Login / Admin Login) using Jinja2 conditional rendering. Login is handled via JavaScript fetch() POST to /login with JSON body. On success, the browser is redirected to the appropriate dashboard. On failure, a Bootstrap danger alert is shown inline without a page reload.

9.3 Student Dashboard — student_dashboard.html

A tabbed Bootstrap interface with three tabs:

  • Attendance tab — Fetches /student/data and renders a table with Subject Code, Date, and Status columns. Status is displayed as a green "Present" or red "Absent" Bootstrap badge.
  • Marks tab — Renders a table with Subject Code and Marks columns from the same API response.
  • Subjects tab — Renders a table with Subject Code, Subject Name, Semester, and Teacher ID columns.

All data is loaded on page load via a single fetch() call. Empty states show a "No records" message in the table body.

9.4 Teacher Dashboard — teacher_dashboard.html

A tabbed Bootstrap interface with three tabs:

  • Mark Attendance tab — Form with Student ID, Subject Code, Status (Present/Absent dropdown), and Date fields. On submit, calls POST /teacher/mark_attendance. Success/error messages auto-dismiss after 3 seconds.
  • Enter Marks tab — Form with Student ID, Subject Code, and Marks (0–100) fields. On submit, calls POST /teacher/enter_marks. Uses upsert so re-submitting updates existing marks.
  • Student List tab — Loads on tab click (lazy loading) via GET /teacher/students. Renders a table with Student ID, Name, Department, and Semester.

9.5 Admin Dashboard — admin_dashboard.html

A tabbed Bootstrap interface with four tabs:

  • Add Student tab — Form with Student ID, Name, Department, and Semester fields. Calls POST /admin/add_student. Auto-creates login on success.
  • Add Teacher tab — Form with Teacher ID, Name, and Department fields. Calls POST /admin/add_teacher. Auto-creates login on success.
  • Assign Subject tab — Form with Subject Code, Subject Name, Semester, and Teacher ID fields. Calls POST /admin/add_subject.
  • View All tab — Two side-by-side tables showing all students and all teachers. Loaded on tab click via parallel Promise.all() fetch calls to /admin/students and /admin/teachers.

All forms show inline success/error alerts that auto-clear after 3 seconds. Forms reset on success.

10. Security Implementation

Security Measure Implementation Detail
Session-Based Authentication Flask server-side sessions store username, role, and user_id after login. A secret key (sarms_secret_key_2024) signs the session cookie to prevent client-side tampering.
Role-Based Dashboard Guards Each dashboard route in app.py checks session.get("role") != "<required_role>" and redirects to / if the check fails. This prevents cross-role access (e.g., a student cannot access the teacher dashboard).
Student Data Isolation The /student/data endpoint uses session.get("user_id") as the filter for attendance and marks queries. Students can only see their own records — they cannot query other students' data.
Student Existence Validation Both mark_attendance() and enter_marks() in teacher_routes query the students_collection to verify the student_id exists before writing any record. Invalid student IDs return HTTP 400.
Session Logout /logout calls session.clear() which removes all session keys, effectively invalidating the login state.
Auto-Login Security Note Auto-created student/teacher accounts use the entity ID as both username and password. This is intentional for a university demo system where the admin distributes credentials. Production systems should enforce password change on first login.

11. Business Logic

11.1 Auto Login Account Creation

When an admin adds a student or teacher, the system automatically creates a login account in the users collection using MongoDB's upsert=True. This means re-adding the same entity updates the existing account rather than creating a duplicate.

# When admin adds student "23BAI001":
users_collection.update_one(
    {"username": "23BAI001"},
    {"$set": {
        "username": "23BAI001",
        "password": "23BAI001",   # default password = student_id
        "role":     "student",
        "ref_id":   "23BAI001"    # stored in session as user_id after login
    }},
    upsert=True
)

11.2 Marks Upsert Logic

Marks are stored using MongoDB's upsert operation. If a mark record already exists for a student+subject combination, it is updated. If not, a new record is created. This prevents duplicate mark entries and allows teachers to correct marks.

marks_collection.update_one(
    {"student_id": data["student_id"], "subject_code": data["subject_code"]},
    {"$set": {"marks": data["marks"]}},
    upsert=True
)
# First call:  creates new document
# Second call: updates existing document's marks field

11.3 Session-Based Data Scoping

The student data API uses the session's user_id (which equals the student_id) to scope all queries. This ensures each student only sees their own data without any additional filtering logic in the frontend.

@student_routes.route("/student/data", methods=["GET"])
def get_my_data():
    student_id = session.get("user_id")   # set during login from ref_id
    if not student_id:
        return jsonify({"error": "Not logged in"}), 401

    attendance = list(attendance_collection.find({"student_id": student_id}, {"_id": 0}))
    marks      = list(marks_collection.find({"student_id": student_id}, {"_id": 0}))
    subjects   = list(subjects_collection.find({}, {"_id": 0}))  # all subjects

    return jsonify({"attendance": attendance, "marks": marks, "subjects": subjects})

11.4 Role-to-Redirect Mapping

After successful login, the system maps the user's role to the appropriate dashboard URL using a Python dictionary. Unknown roles default to the home page.

redirects = {
    "student": "/student_dashboard",
    "teacher": "/teacher_dashboard",
    "admin":   "/admin_dashboard"
}
return jsonify({"redirect": redirects.get(user["role"], "/")})

15. Limitations & Future Enhancements

15.1 Current Limitations

# Limitation Impact
1 Passwords stored as plain text in MongoDB Security risk — passwords should be hashed with bcrypt or SHA-256
2 No admin authentication guard on admin API routes Admin APIs (/admin/add_student etc.) are accessible without login session check
3 Default password equals student/teacher ID Weak default credentials; no forced password change on first login
4 No attendance percentage calculation Students see raw records but no summary (e.g., 75% attendance in CS301)
5 No subject enrollment — all subjects shown to all students Students see subjects from all semesters, not just their enrolled ones
6 No duplicate attendance check Teacher can mark attendance for the same student+subject+date multiple times
7 No pagination on student/teacher lists Large datasets may cause slow page loads in admin and teacher views
8 No input validation on admin forms beyond HTML5 required Invalid data (e.g., duplicate student_id) can be inserted without error feedback

15.2 Future Enhancements

# Enhancement Description
1 Password Hashing Implement bcrypt or SHA-256 hashing for all stored passwords
2 Admin Route Guards Add session role check to all admin API endpoints to prevent unauthorized access
3 Attendance Summary Calculate and display attendance percentage per subject for each student
4 Subject Enrollment Link students to specific subjects based on semester and department
5 Duplicate Attendance Prevention Check for existing attendance record before inserting (same student+subject+date)
6 Grade Calculation Auto-calculate letter grades (A/B/C/D/F) from marks and display in student dashboard
7 Result Report Card Generate a printable/downloadable PDF report card for each student
8 Email Notifications Send email to students when marks are entered or attendance is marked
9 Admin Analytics Dashboard charts showing department-wise attendance rates and average marks
10 Mobile Responsive Improvements Optimize tab layouts and tables for small screen devices

16. Conclusion

The Student Attendance & Result Management System (SARMS) successfully delivers a complete, functional university portal that digitizes the core academic record-keeping processes. The system supports three distinct user roles — Student, Teacher, and Admin — each with a dedicated, role-protected dashboard built with Bootstrap 5 tabs for a clean and intuitive user experience.

The Flask Blueprint architecture ensures clean separation of concerns across authentication, student data, teacher operations, and admin management. MongoDB's flexible document model accommodates the varied data structures across students, teachers, subjects, attendance records, and marks without complex relational joins. The auto-login creation feature significantly reduces admin overhead by automatically provisioning credentials when new students or teachers are added.

Key technical achievements include session-based role-based access control protecting all three dashboards, MongoDB upsert for idempotent marks entry, session-scoped student data isolation ensuring privacy, and a comprehensive seed script providing realistic demonstration data across all collections.

The project was validated through 73 automated test cases (30 black box + 43 white box) with a 100% pass rate, confirming the correctness of all schemas, guard logic, API responses, and data persistence operations.

With the identified enhancements — particularly password hashing, admin route guards, attendance percentage calculation, and subject enrollment — SARMS can evolve from a demonstration system into a production-grade university management platform.
