1. Objective

  • To implement automation testing using Selenium WebDriver for web-based applications
  • To convert manual test cases into automated, repeatable scripts using Python and pytest
  • To validate system functionality including login, registration, and core module operations
  • To cover functional validation, input handling, and error scenarios automatically
  • To demonstrate the importance of automation in modern software development workflows
  • To automate both the Login Module and Core Modules of SARMS and SRTBS projects

2. Problem Statement

Manual testing of web applications is time-consuming, error-prone, and difficult to repeat consistently across multiple test cycles. As applications grow in complexity, the number of test cases increases, making manual execution impractical for regression testing.

This experiment develops and executes automated test cases for the SARMS (Student Attendance & Result Management System) and SRTBS (Smart Railway Ticket Booking System) using Selenium WebDriver. The automation covers:

  • Functional validation of login and registration flows for all user roles
  • Input handling — valid inputs, invalid inputs, empty fields
  • Error scenario testing — wrong credentials, duplicate registration, unauthorized access
  • Core module operations — attendance marking, marks entry, train search, booking, admin management
  • Session management — login state, logout, direct URL access without authentication

5. Automation Architecture

selenium_tests/
│
├── test_sarms_selenium.py
│       ├── @pytest.fixture(scope="class") driver  ← Chrome WebDriver setup/teardown
│       ├── TestSARMSHomePage      (5 tests)  ← Home page navigation
│       ├── TestSARMSLoginModule   (13 tests) ← Login/logout all 3 roles + error cases
│       ├── TestSARMSStudentCore   (8 tests)  ← Student dashboard tabs + data
│       ├── TestSARMSTeacherCore   (7 tests)  ← Attendance + marks + student list
│       └── TestSARMSAdminCore     (9 tests)  ← Add student/teacher/subject + view all
│
└── test_srtbs_selenium.py
        ├── @pytest.fixture(scope="class") driver  ← Chrome WebDriver setup/teardown
        ├── TestSRTBSHomePage      (5 tests)  ← Home page + navbar links
        ├── TestSRTBSRegistration  (4 tests)  ← Register valid/duplicate/empty
        ├── TestSRTBSLoginModule   (11 tests) ← Login/logout user+admin + error cases
        ├── TestSRTBSTrainSearch   (6 tests)  ← Search valid/invalid routes + fare
        ├── TestSRTBSBookingCore   (4 tests)  ← My bookings + auth check
        └── TestSRTBSAdminCore     (7 tests)  ← Add train + bookings/users tabs

Each test class uses a shared driver fixture (scope="class") so the browser is opened once per class and closed after all tests in that class complete. Helper methods (_login_student, _fill_login, etc.) reduce code duplication across tests.

9. Sample Automation Code

9.1 WebDriver Setup (Fixture)

@pytest.fixture(scope="class")
def driver():
    options = webdriver.ChromeOptions()
    options.add_argument("--start-maximized")
    options.add_argument("--disable-notifications")
    drv = webdriver.Chrome(
        service=Service(ChromeDriverManager().install()),
        options=options
    )
    drv.implicitly_wait(6)   # wait up to 6s for elements
    yield drv
    drv.quit()               # teardown: close browser after class

9.2 Login Test — Valid Credentials

def test_SA_LM_02_valid_student_login(self, driver):
    """SA-LM-02: Valid student credentials redirect to student dashboard."""
    driver.get("http://127.0.0.1:5000/login/student")
    time.sleep(0.5)

    # Fill username and password
    driver.find_element(By.ID, "username").send_keys("23BAI001")
    driver.find_element(By.ID, "password").send_keys("23BAI001")

    # Click submit button
    driver.find_element(By.CSS_SELECTOR, "button[type='submit']").click()
    time.sleep(1.5)

    # Assert redirect to student dashboard
    assert "student_dashboard" in driver.current_url

9.3 Login Test — Invalid Credentials

def test_SA_LM_05_invalid_credentials(self, driver):
    """SA-LM-05: Invalid credentials show error message."""
    driver.get("http://127.0.0.1:5000/login/student")
    time.sleep(0.5)

    driver.find_element(By.ID, "username").send_keys("wronguser")
    driver.find_element(By.ID, "password").send_keys("wrongpass")
    driver.find_element(By.CSS_SELECTOR, "button[type='submit']").click()
    time.sleep(1.5)

    # Assert error message is shown
    page = driver.page_source
    assert "Invalid" in page or "error" in page.lower()

9.5 SRTBS — Train Search

def test_SR_TS_03_valid_search_returns_results(self, driver):
    """SR-TS-03: Valid route search returns train results."""
    driver.find_element(By.ID, "source").send_keys("Mumbai")
    driver.find_element(By.ID, "destination").send_keys("Delhi")
    driver.find_element(By.ID, "journey_date").send_keys("2026-12-01")

    # Click search button
    driver.find_element(By.CSS_SELECTOR, "button[onclick='searchTrains()']").click()
    time.sleep(2)   # wait for fetch() API call to complete

    # Assert results contain train data
    page = driver.page_source
    assert "Rajdhani" in page or "12951" in page or "Book" in page

9.6 Explicit Wait Usage

from selenium.webdriver.support.ui import WebDriverWait
from selenium.webdriver.support import expected_conditions as EC

def wait_for(driver, by, value, timeout=6):
    """Wait for element to be present in DOM."""
    return WebDriverWait(driver, timeout).until(
        EC.presence_of_element_located((by, value))
    )

# Usage:
element = wait_for(driver, By.ID, "username")
element.send_keys("23BAI001")

10. Advantages of Automation Testing

# Advantage Description
1 Speed Automated tests execute in seconds what would take minutes manually. 79 tests run in ~128 seconds vs. hours of manual testing.
2 Repeatability Tests produce identical results every run, eliminating human inconsistency in test execution.
3 Regression Testing After any code change, the full test suite can be re-run instantly to detect regressions.
4 Parallel Execution Multiple test classes can run simultaneously using pytest-xdist, further reducing execution time.
5 24/7 Availability Automated tests can run at any time including nights and weekends without human presence.
6 Accurate Reporting pytest generates detailed pass/fail reports with exact assertion failures and stack traces.
7 Cross-Browser Testing Selenium supports Chrome, Firefox, Edge, and Safari — the same scripts can test multiple browsers.
8 CI/CD Integration Selenium tests can be integrated into GitHub Actions, Jenkins, or GitLab CI for automatic testing on every commit.

11. Limitations

# Limitation Impact
1 Requires running application Flask app and MongoDB must be running before tests execute
2 Browser dependency Tests require Chrome and ChromeDriver; version mismatches can cause failures
3 Timing sensitivity JavaScript-heavy pages require explicit waits; insufficient wait times cause flaky tests
4 No headless mode configured Tests open a visible browser window; headless mode can be added with --headless flag
5 Test data dependency Tests depend on seed data (23BAI001, TCH001, admin) being present in MongoDB
6 No visual regression testing Selenium validates functionality but not visual appearance or CSS rendering
7 Maintenance overhead UI changes (element IDs, class names) require updating test selectors

12. Conclusion

This experiment successfully implemented Selenium WebDriver automation testing for both the SARMS and SRTBS web applications. A total of 79 automated test cases were developed across 11 modules covering the Login Module and Core Modules of both systems.

The automation scripts validate all critical user flows including role-based login for all three SARMS roles (Student, Teacher, Admin) and both SRTBS roles (User, Admin), error handling for invalid credentials and empty fields, session management and unauthorized access prevention, and core operations such as attendance marking, marks entry, train search, booking history, and admin management.

Using Python pytest with Selenium WebDriver and webdriver-manager provides a clean, maintainable automation framework. The class-scoped driver fixture ensures efficient browser resource usage, while helper methods reduce code duplication. Explicit waits handle asynchronous JavaScript fetch calls reliably.

The automation demonstrates the key advantage of repeatable, fast regression testing — what would require hours of manual testing across multiple roles and scenarios executes automatically in approximately 2 minutes, making it practical to run after every code change.
